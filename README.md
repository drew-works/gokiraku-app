# 店舗インフラと人手不足の課題を解決する「オフライン型」注文管理システム

アルバイト先のお好み焼き・もんじゃ焼き店「ごきらく亭」が抱える「通信環境の悪さ」と「常態化するワンオペ」という実現場の課題を解決するために開発した、Web注文システムです。

##プロジェクトの全体像
本システムは、インターネット環境に依存せず、現場の業務動線をスムーズにする2つの機能で構成されています。

| 画面・ファイル名 | 対象ユーザー | 役割・主な機能 |
| :--- | :--- | :--- |
| **`order.html`**<br>`order-script.js`<br>`order-style.css` |　 **お客様用**<br>(スマホ画面) 　| 卓番号の自動判定、トッピング選択、カゴ管理、オフラインQRコード生成機能 |
| **`staff.html`**<br>`staff-logic.js`<br>`staff-scanner.js` |　 **店員・厨房用**<br>(店舗端末)  | QRコードスキャン、全卓共通キッチン未提供リスト、各卓の注文履歴・2段階ステータス管理 |
| **`order-menu-data.js`** |  **データマスタ**  | 全画面共通のメニュー名・価格・トッピングデータの管理 |

## ✨ 現場の課題解決に向けた設計のポイント

### 1. ワンオペ時の手書き伝票による「機会損失」の撲滅
- **背景と課題：**
  当店では人手不足が深刻で、週末の混雑時であってもホールを1人で回さなければならない日があります。
  満席時にお客様から一斉に口頭で注文を受けると、手書き伝票への記入や記憶の保持が追いつかず、オーダーと料理の提供が物理的に遅れていました。その結果、客席や厨房の調理キャパシティにはまだ余裕があるにもかかわらず、「ホールの注文処理が追いつかない」という理由で新規のお客様をお断りせざるを得ない深刻な売上のロスが多発していました。
- **アプローチと効果：**
  注文処理をデジタル化し、お客様が選んだ正確な注文データが店員側に一瞬で届く仕組みに改善しました。これにより、注文の聞き取りやメモにかかっていた時間を「料理の提供や接客」に集中させることが可能になり、ワンオペ時でもお客様を断らずにスムーズに迎え入れられる体制を作りました。

### 2. 店舗の通信制限をクリアする「オフライン対応QRコード」
- **背景と課題：** 一般的なQR注文システムは、お客様のスマホがインターネットに繋がることが大前提です。しかし、当店は立地的に電波が入りにくく、店舗へのWi-Fi導入も予算的に困難という致命的なインフラの壁がありました。
- **アプローチと効果：** ネットワーク通信を一切挟まず、**お客様のスマホのローカル環境で注文データを文字列に暗号化し、その場でQRコードとして生成するロジック**を実装しました。店員の端末カメラでそのQRを直接読み取る仕様にしたため、通信環境がどれだけ悪くても確実に機能する注文のデジタル化を実現しました。

### 3. 接客の手間引き算と、無駄な待機時間の削減
- **背景と課題：** 混雑時にお客様に呼ばれてテーブルへ向かった際、「まだ注文が決まっていない」と言われ、その場で数分間待たされてしまうタイムロスが頻発していました。1分1秒を争うワンオペ環境において、この拘束時間は大きな負担でした。
- **アプローチと効果：** お客様が手元でメニューを確定させてから店員を呼び、QRを提示する流れにしたことで、スタッフの無駄な拘束時間を完全に排除し、店舗全体の稼働効率を向上させました。

### 4. アットホームなお店の強み（接客・会話）を残す仕様
- **背景と課題：** 完全にスマホだけで決済まで完結するシステムは業務を効率化する一方、お店が大切にしている「お客様とのアットホームな会話や温かみ」をなくしてしまいます。
- **アプローチと効果：** あえて「店員がお客様のテーブルに出向き、スマホのQRコードを直接スキャンする」という対面ステップを1つ残しました。これにより、注文作業自体の時間は大幅にカットしつつも、お客様と店員が必ず顔を合わせる「接客の余白」を大切にしています。

## 運用テストを経て実施した仕様改善（PDCA）

実際に店舗でテスト運用を重ね、店員側の動線やミスの傾向に合わせて自ら課題を発見・修正した内容です。

### A. 「📋 全卓共通」キッチン未提供リストの追加（ `staff-logic.js` ）
* **見つかった課題：** 当初は「卓ごとの注文履歴」しか見られず、厨房（キッチン）のスタッフが「今、店全体でどの料理を最優先で作るべきか」を一目で判断できない問題がありました。
* **改善：** 選択タブの一番左端に**「全卓共通タブ」**を新設。全11卓の未提供注文だけを抽出し、**注文が入った時刻順（古い順）に自動で一列に並べる仕様**にアップデートしました。共通画面で提供完了を押すと、リストから即座に消去されると同時に、各卓の個別履歴データ側も連動して自動で「提供済」に切り替わるようにしました。

### B. トッピング判定のバグ修正による視認性の向上
* **見つかった課題：** 調理が未完了の卓を赤くハイライトする仕様にしていましたが、メイン料理の調理が終わっても「追加トッピング（エビやイカなど）」の裏データが未完了扱いとして残ってしまい、卓の赤みが消えないバグを発見しました。
* **改善：** トッピングのID判定ロジックを修正し、色変えの判定フラグからトッピングデータを正確に除外。メイン料理の提供完了に合わせて正しく卓の色が切り替わるようにロジックを最適化しました。

## 💻 フロントエンド開発における実装の工夫

- **URLパラメータによる「卓番号の自動判定」（ `order.html` ）：**
  卓ごとに専用のQRコードを用意し、アクセス時に `?table=3` などのパラメータを読み込ませる設計にしました。お客様が自分で卓番号を入力する手間を完全に排除しています。

- **オペレーションの手数を減らす「自動画面切り替え」（ `staff-scanner.js` ）：**
  ホールスタッフが忙しいときでも操作迷子にならないよう、お客様のQRコードを読み取った瞬間に、システムが自動で「その注文があった卓の個別ページ」へ画面を遷移させるロジックをJavaScriptで実装しました。

- **ミスを防ぐ「伝票」と「提供」の2段階ステータス管理（ `staff-logic.js` ）：**
  お好み焼き店でのミスの原因を分析し、「注文を紙の伝票に書き写すステップ」と「料理をテーブルへ運ぶステップ」の2ヶ所でトリガーがあることに着目。店員側アプリに「伝票 📝」と「提供 🍳」の独立したボタンを設け、それぞれの完了状態を色（赤/緑）で視覚的に判別できるようにしました。

## 今後の展望

実際の現場運用を想定した際、今後さらにブラッシュアップしていきたいと考えている課題です。

1. **複数端末でのデータ同期**
   - **課題：** ローカルストレージにデータを保存しているため、ホールが2人いた際、リアルタイムにデータを共有できず、ミスにつながる。
   - **対策：** インターネット通信は使わず、店舗内に安価なローカルルーターを1台設置し、WebSocketなどを利用して「店内の閉じたネットワーク内だけで複数端末が同期する仕組み」へ拡張したいです。

3. **清算時（お会計時）の自動合計金額算出機能**
   - **課題：** 現状は「卓を清算」ボタンでデータを消去するのみで、実際のレジ打ちの手間はまだ残っています。
   - **対策：** 個別卓の履歴画面に、トッピング等も含めたその卓の「合計金額」を自動計算して一目で確認できるエリアを実装し、レジ業務のさらなる時間短縮に繋げたいです。

## 🛠️ 使用技術・開発手法
- **言語・ツール：** HTML5 / CSS3、JavaScript 
- **外部ライブラリ：** `jsQR.min.js`（カメラ映像からのQRコード解析）、`qrcode.min.js`（お客様側のQRコード生成）
- **開発プロセスにおける工夫：**
  今回のオフラインでの注文データ受け渡しロジックの構築や、データの肥大化を防ぐためのリファクタリングにおいて、AIを開発パートナーとして活用。設計の整合性やコードの最適化について密にディスカッションを重ねながら実装を進めました。
