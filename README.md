# 人手不足と店舗インフラの課題を解決する「ハイブリッド型」注文決済システム
実際のアルバイト先（お好み焼き・もんじゃ焼き店「ごきらく亭」）が抱える「深刻な人手不足」と「店舗インターネットの脆弱さ」というリアルな現場課題を解決するために開発した、新しいWeb注文システムです。

## ✨ 本プロジェクトにおける最大のこだわり（仕様改善・UX）

### 1. 店舗のインフラ制限を逆手に取った「オフライン対応QRコード」設計
- **現場の課題：** 一般的なQRコード注文は、お客様のスマホがインターネットに繋がることが前提です。しかし、丘の下にあることからスマホがすぐ圏外になってしまい、WIFIを導入するのもオーナーが躊躇っているため、諦めました。
- **技術的な工夫と仕様改善：** ネット通信を必須とせず、**お客様のスマホ画面上で注文を確定させ、その場で注文内容が含まれた「QRコード」を生成する仕組み**にしました。店員側がそのQRをスキャンして伝票に書き写す形にすることで、店舗の電波状況に関係なく、確実に注文のデジタル化・時間短縮を実現しました。

### 2. アルバイトの経験から生まれた「業務効率化」と「注文ミス防止」
- **口頭注文と手書き伝票による注文ミスの撲滅：**
  これまでは、お客様から口頭で言われた注文をその場で紙の伝票に手書きしていました。しかし、別の作業中で伝票を持っていない瞬間にたくさんの注文を一気に言われると、忙しい混雑時には記憶が追いつかず、結果として「提供忘れ（注文ミス）」を起こしてしまう課題がありました。本システムによってお客様が選んだデータがそのままデジタルで店員側に届くため、人間の記憶力に頼らない確実なオペレーションへ仕様改善しました。
- **機会損失と無駄な待機時間の削減：** 
  混雑時にお客様に呼ばれてテーブルへ向かったものの、「まだ注文が決まっていない」と言われ、その場で3分ほど待たされてホールの貴重な時間をロスすることがよくありました。システム化により、お客様が手元でじっくり選んでから店員を呼ぶ流れになったため、スタッフが1人しかいない時間帯でも、無駄な待機時間を一切なくして効率的に動けるようになりました。

### 3. デジタル化しても「お店の強み（接客・会話）」を消さない温かい設計
- **あえて完全非対面にしない選択：**
  完全にスマホだけで注文・決済が完結するシステムは、人手不足を解消する一方で、お店が大切にしている「お客様とのアットホームな会話」を消してしまいます。
  本システムでは、あえて「店員がお客様のスマホのQRをスキャンする」という1ステップを挟むことで、**業務を圧倒的に効率化しつつも、お客様と店員が必ず顔を合わせ、ちょっとした会話や温かみが生まれる接客の「余白」**を残しました。

  ### 4. 💻 フロントエンド開発における実装の工夫

- **URLパラメータを活用した「卓番号の自動判定」設計（ソース1）：**
  お客様が座る席（卓）ごとに専用のQRコードを用意し、そこから `?table=3` のようなURLを開かせる設計にしました。これにより、システム側で自動的に「〇番卓のお客様からの注文」と判別できるため、お客様がいちいち卓番号を入力する手間を徹底的に排除しました。

- **店員側の負担を極限まで減らす「自動画面切り替え」機能（ソース2）：**
  ホールがワンオペの時でも焦らず使えるよう、店員がお客様のスマホのQRコードをカメラで読み取った瞬間、システムが自動的に「注文があった卓のページ」へ画面をパッと切り替えるロジックをJavaScriptで実装しました。これにより、店員がわざわざ手動で卓のボタンを探して切り替える手数を減らし、爆速で調理や伝票記入に移れるようにしました。

- **「伝票未/済」と「提供未/済」の2段階ステータス管理（ソース2）：**
  お好み焼き屋のリアルな現場では、「注文を紙の伝票に書き写すステップ」と「料理をテーブルに運ぶステップ」の2つのミスが起きやすいです。これを防ぐため、店員アプリ側には「伝票 📝」と「提供 🍳」のボタンを別々に用意し、それぞれの完了状態を色（赤/緑）でひと目で確認できるようにしました。

## 🛠️ 使用技術
- HTML5 / CSS3（レスポンシブ対応）
- JavaScript（注文データの保持、QRコード生成ロジックの制御）
- **AI-Driven Engineering：**
  「Wi-Fiが使えない環境でどうやってデジタル注文を実現するか」「接客の温かみをどう残すか」という要件定義から、それを実現するロジック（JavaScriptでのQR生成など）への落とし込みをAIと共創しました。
