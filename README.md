# 人手不足と店舗インフラの課題を解決する「ハイブリッド型」注文決済システム

私のアルバイト先（お好み焼き・もんじゃ焼き店）が抱える「深刻な人手不足」と「店舗インターネットの脆弱さ」というリアルな現場課題を解決するために開発した、新しいWeb注文システムです。

## 📋 プロジェクト概要（システム構成）
本システムは、ネット環境に依存せず、店舗運営を円滑にする2つの画面で構成されています。

| 画面・ファイル名 | 対象ユーザー | 役割・主な機能 |
| :--- | :--- | :--- |
| **`order.html`**<br>`order-script.js`<br>`order-style.css` | **お客様用**<br>(スマホ画面) | 卓番号の自動判定、トッピング選択、カゴ管理、**オフラインQRコード生成機能** |
| **`staff.html`**<br>`staff-logic.js`<br>`staff-scanner.js` | **店員・厨房用**<br>(店舗端末) | QRコードスキャン、**全卓共通キッチン未提供リスト**、各卓の注文履歴・2段階ステータス管理 |
| **`order-menu-data.js`** | **データマスタ** | 全画面共通のメニュー名・価格・トッピングデータの管理 |

---

## ✨ 本プロジェクトにおける最大のこだわり（現場の課題解決とUX）

### 1. 深刻な人手不足と、手書き伝票による「機会損失」の撲滅
- **現場の課題：**
  当店では人手不足が深刻化しており、**ピーク時であっても基本的にはホールをワンオペで回さなければならない**という過酷な現状がありました。
  そのため、満席時にお客様から一斉に口頭で注文を言われると、手書き伝票への記入や記憶の限界を超えてしまい、オーダーと料理の提供が物理的に間に合わなくなっていました。結果として、**席は空いている、あるいは料理を作るキャパシティはあるのに、「ホールが回らないから」という理由で新規のお客様入店を断らざるを得ないという、深刻な機会損失（**が多発していました。
- **技術的な工夫と仕様改善：**
  本システムにより、お客様が選んだ正確な注文データがデジタルで一瞬で店員側に届くため、人間の記憶力や手書きの手間に頼らない確実なオペレーションへ変革しました。これにより、注文対応にかかっていたホールの時間をドリンク作成や提供へ集中させることが可能となり、ワンオペ時でもお客様を断らずに迎え入れられる体制を構築しました。

### 2. 店舗のインフラ制限を逆手に取った「オフライン対応QRコード」設計
- **現場の課題：** 一般的なQR注文は顧客のスマホのネット接続が前提ですが、当店は立地的に電波が弱く、Wi-Fiの導入も予算的に困難という致命的なインフラの課題がありました。
- **技術的な工夫と仕様改善：** 通信を一切挟まず、**お客様のスマホのブラウザ上だけで注文データを暗号化し、その場で注文内容が含まれた「QRコード」を即座に生成する仕組み（ローカル処理）**を実装しました。店員側がそのQRをスキャンしてデータを読み取るため、店舗の電波状況に関係なく、確実に注文のデジタル化を実現しました。

### 3. 機会損失と無駄な待機時間の削減
- **現場の課題：** 混雑時にお客様に呼ばれてテーブルへ向かったものの、「まだ注文が決まっていない」と数分間その場で拘束され、他の卓の対応がすべてストップしてしまうホールのタイムロスが多発していました。ワンオペ環境において、この数分間の拘束は致命傷となります。
- **技術的な工夫と仕様改善：** システム化により、手元でじっくり選んでから店員を呼ぶ流れになり、ワンオペの時間帯でも無駄な待機時間を一切なくして、1人のスタッフが極限まで効率的に動けるようになりました。

### 4. デジタル化しても「お店の強み（接客・会話）」を消さない温かい設計
- **あえて完全非対面にしない選択：**
  完全にスマホだけで注文・決済が完結するシステムは、人手不足を解消する一方で、お店が大切にしている「アットホームな距離感」を消してしまいます。本システムでは、あえて「店員がお客様のスマホのQRをスキャンする」という1ステップを挟むことで、**業務を圧倒的に効率化しつつも、お客様と店員が必ず顔を合わせ、ちょっとした会話や温かみが生まれる接客の「余白」**を残しました。

## 🚀 現場でのテストを経て自ら導入した「さらなる仕様改善」（PDCA）

実際にシステムを店舗でテスト運用し、現場の動線に合わせて自ら課題を発見・実装した**発展的アップデート**です。

### A. 「📋 全卓共通」キッチン未提供リストの創出（ `staff-logic.js` ）
* **現場で気づいた課題：** 初期状態では「卓ごとの履歴」しか見られず、調理担当（キッチン）が「今、店全体で何を優先して作ればいいのか」を一目で把握できない問題がありました。
* **解決策：** タブの一番左端に**「全卓共通タブ」**を新設。全11卓の「未提供 🍳」の料理だけを、**注文が入った時刻順（古い順）に自動で一列に並べる厨房専用の司令塔画面**へと進化させました。この画面で提供完了を押すと、リストから消えると同時に、各卓の個別データも自動で「提供済」に連動します。

### B. トッピングデータの裏側判定による「卓の色変えバグ」の修正
* **現場で気づいた課題：** メイン料理が未提供の卓を赤く光らせる仕様にしていましたが、「トッピング（エビやイカなど）」が裏データとして未提供扱いとして残ってしまい、料理を提供し終わっても卓の赤みが消えないバグを発見しました。
* **解決策：** トッピングのID判定ロジックを見直し、色変えのトリガーからトッピングデータを正確に除外。メイン料理の完了だけで正しく看板の色が切り替わるようロジックを最適化し、現場の誤認を防ぎました。

## 💻 フロントエンド開発における実装の工夫

- **URLパラメータを活用した「卓番号の自動判定」設計（ `order.html` ）：**
  お客様が座る席（卓）ごとに専用のQRコードを用意し、そこから `?table=3` のようなURLを開かせる設計にしました。これにより、システム側で自動的に「3番卓のお客様からの注文」と判別できるため、お客様がいちいち卓番号を入力する手間を徹底的に排除しました。

- **店員側の負担を極限まで減らす「自動画面切り替え」機能（ `staff-scanner.js` ）：**
  ホールがワンオペの時でも焦らず使えるよう、店員がお客様のスマホのQRコードをカメラで読み取った瞬間、システムが自動的に「注文があった卓のページ」へ画面をパッと切り替えるロジックをJavaScriptで実装しました。

- **「伝票未/済」と「提供未/済」の2段階ステータス管理（ `staff-logic.js` ）：**
  お好み焼き屋のリアルな現場では、「注文を紙の伝票に書き写すステップ」と「料理をテーブルに運ぶステップ」の2つのミスが起きやすいです。これを防ぐため、店員アプリ側には「伝票 📝」と「提供 🍳」のボタンを別々に用意し、それぞれの完了状態を色（赤/緑）でひと目で確認できるようにしました。

## 🛠️ 使用技術・開発手法
- **フロントエンド：** HTML5 / CSS3（レスポンシブ対応）
- **ロジック：** JavaScript (ES6) 
- **外部ライブラリ：** `jsQR.min.js`（カメラ映像からのQR高速解析）、`qrcode.min.js`（顧客側QR生成）
- **AI-Driven Engineering（AIとの共創）：**
  「Wi-Fiが使えない環境でどうやってデジタル注文を実現するか」「接客の温かみをどう残すか」という現場の要件定義から、それを実現する高度なJavaScriptのロジック構築、データ分割、リファクタリングに至るまで、AIを開発パートナーとして徹底的にディスカッションを重ねながら共創しました。
